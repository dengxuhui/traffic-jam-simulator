## 代码风格与规范
以下规范基于现有代码风格与本仓库的实际规模。

### 通用约定
- 尽量保持单文件结构，除非明确需要拆分
- 避免引入构建工具或依赖，除非需求明确
- 保持整体风格紧凑、性能优先
- 保持中文 UI 文案与中文注释风格一致

### HTML
- 语义化标签优先
- 保持结构层级清晰，避免无意义嵌套
- `id` 用于交互绑定，`class` 用于样式
- DOM 结构应保持与现有布局风格一致

### CSS
- 使用 `:root` 变量集中管理颜色与基础尺度
- 颜色变量命名简短（如 `--bg`、`--tx`）
- 避免重复硬编码颜色，优先复用变量
- 采用紧凑写法（现有文件中多为单行规则）
- 动效保持轻量（`transition: all .1s` 类似）
- 保持暗色主题与现有视觉一致

### JavaScript
- 仅使用原生 JS，不引入框架
- 变量命名与现有风格保持一致（短小但可读）
- 数据结构保持轻量，不创建过深层级
- 函数尽量短小、直接，避免复杂抽象
- 避免使用 `eval` 或动态代码执行

### 导入与依赖
- 当前不使用模块系统（无 `import`/`export`）
- 不新增第三方依赖，除非明确要求

### 格式化
- 现有文件为压缩式排版，保持现有风格
- 只在必要时换行以提升可读性
- 保持同一块逻辑的风格一致

### 类型与数据
- 无 TypeScript
- 若新增复杂数据结构，请用注释说明字段含义
- 维持数值单位一致（例如 km/h、m/s）

### 命名规范
- `id`：简短并与功能一致（如 `avgS`、`den`）
- `class`：短写但语义明确（如 `s1`、`cb`）
- 变量：短名为主，但避免歧义
- 常量使用 `const`，可变使用 `let`

### 错误处理
- 交互逻辑优先保证健壮性（空值检查）
- DOM 查询后如果可能为空，先做判空
- 尽量避免抛出异常影响主循环

### 性能约定
- 动画主循环使用 `requestAnimationFrame`
- 更新频率与计算逻辑应保持轻量
- 任何新增逻辑需避免在每帧重复创建大量对象

## 交互与可视化
- 坐标系、速度、加速度等单位需保持一致
- 车辆绘制逻辑应保持一致的视觉风格
- HUD 文案与颜色保持一致

## 文件新增/修改建议
- 若新增文件，应在 `index.html` 中适配引用
- 保持总文件数少，避免拆分过度
